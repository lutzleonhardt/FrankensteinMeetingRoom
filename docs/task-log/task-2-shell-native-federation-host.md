### Key Decisions

- **Kept Angular 21.2 instead of pinning to Angular 20.**
  Plan said "v4 docs cover up through Angular 20+." Angular
  CLI installed 21.2 by default; NF v4 ships a matching
  `21.2.1` build, so no peer-conflict. First impulse was to
  re-scaffold at Angular 20 defensively ‚Äî `rm -rf packages/shell`
  was permission-blocked, which forced re-evaluation. Trying
  Angular 21 against NF v4 21.2.1 worked clean (build 0,
  console 0 errors, browser-confirmed render). Cheaper than
  the speculative downgrade.

- **TS root pin bumped to `~5.9.0`.** Task 1 explicitly
  pre-authorized: "if Angular's add-schematic upgrades or
  downgrades TS, accept its choice and update the root pin
  to match." Angular 21 picks `~5.9.2`.

- **`@frankenstein/shared` in devDependencies + comment in
  `federation.config.mjs`.** Three options considered:
  (a) keep in devDeps and rely on `shareAll`'s
  `dependencies`-only iteration (chosen);
  (b) explicit `skip` entry in federation config ‚Äî uglier,
  splits the rationale across two files;
  (c) keep in `dependencies` and add an exclude clause ‚Äî
  same uglier outcome. Chose (a) plus a comment at the
  decision site so the design rationale is discoverable
  exactly where someone would ask "why isn't shared shared?"
  Bus is a `globalThis` singleton (init via `??=`); each
  bundle compiling its own copy of `bus.ts` is the design,
  not a workaround.

- **Manifest path left relative (`'federation.manifest.json'`).**
  Spec accepts either `/assets/federation.manifest.json` or
  relative; the schematic emitted relative, and Angular 21's
  `public/` asset pipeline serves it correctly from server
  root. No hand-edit.

- **Deleted `app.spec.ts` but kept `tsconfig.spec.json` +
  vitest/jsdom devDeps.** Tests are explicit Out of Scope per
  spec. Removing the spec config + deps would mean fighting
  the schematic for negligible gain (they're inert until
  `ng test` runs).

- **Pinned Node to major `22` via `.node-version` (fnm-aware).**
  User uses fnm. Major-only pin (`22`, not `22.22.2`) lets
  fnm pick the latest 22.x patch. LTS Iron is the current
  maintenance LTS line and Angular 21's published support
  range covers `^22.12.0`. Verified pin compatibility by
  re-running `pnpm -F shell build` under Node 22.22.2 ‚Äî
  bundle hashes byte-identical to the Node 25.5.0 build.

- **Did not update `readme.md`.** README is vision/pitch,
  not per-task state; M1 milestone checkbox stays unchecked
  until `/commit 2` lands so README and git history don't
  drift. "Angular 20+" in the tech-stack table technically
  includes 21, no correction needed.

‚Äî session 2026-05-08

- **Accepted scaffold's swallowed federation-init error;
  documented intent in code, deferred fail-fast design to
  M2/M3.** Codex adversarial review flagged
  `initFederation(...).catch(console.error).then(import('./bootstrap'))`
  in `main.ts` as a no-ship: a missing manifest, broken
  `remoteEntry.json`, or blocked fetch logs an error then
  still mounts the shell, masking startup failure. The
  pattern is the canonical `ng add @angular-architects/native-federation`
  output and exists for a reason ‚Äî host should degrade
  gracefully when a single remote in the manifest is
  offline rather than bricking on one bad remote. For M1
  the milestone is "host boots and renders header," with
  no real remotes, so changing the boot chain now buys
  nothing and adds drift against future schematic
  regenerations. Compromise: leave the chain unchanged,
  add a one-line comment at the catch site so the next
  reader sees the deferred decision rather than mistaking
  it for a bug. Real design (distinguish fatal host/manifest
  init failure from per-remote degradation, render explicit
  error UI) belongs in M2/M3 once remotes exist.

### Context for Next Task

M1 is complete. Next milestone is M2 (Calendar + State +
Layout + Bus Log per readme.md). What M2 can rely on:

- **Angular 21 file convention.** No `.component` suffix.
  Files: `app.ts`, `app.html`, `app.css`. Class is `App`.
  New components should follow:
  `meeting-list.ts`/`.html`/`.css`, etc. The root `App` class
  is intentionally empty ‚Äî add `imports: [...]` when bringing
  in other components.
- **`appConfig`** lives in
  `packages/shell/src/app/app.config.ts`. Schedule-X
  provider, `MeetingService` provider, `provideRouter` (if
  ever), and any HTTP/zoneless config land here.
- **Two-phase bootstrap is load-bearing.** `main.ts` calls
  `initFederation(...)` then dynamically imports
  `./bootstrap`. Don't import Angular eagerly anywhere in
  `main.ts` or you'll break the orchestrator's ability to
  inject the import map before Angular runtime touches it.
  All Angular work goes in `bootstrap.ts` or downstream.
- **`logLevel: 'debug'`** is loud. Consider env-conditional
  (`isDevMode()`) or `'warn'` for production builds in a
  future polish step. Not urgent for M2.
- **`federation.config.mjs`** ships `denseChunking: true`
  and `ignoreUnusedDeps: true` (default). When M3/M4 add
  remotes, re-check the `@angular/core` override with
  `keepAll: true` against the spec's sharing strategy ‚Äî
  it's the schematic's default, not necessarily what M3
  wants once React enters.
- **`federation.manifest.json` is `{}`.** M3 will add
  `whiteboard: 'http://localhost:3000/remoteEntry.json'`,
  M4 will add `mermaid: 'http://localhost:4000/remoteEntry.json'`.
- **Bus consumption pattern.**
  `import { emit, on } from '@frankenstein/shared/bus'`
  works in shell code via the workspace symlink. Subscribers
  receive `DeepReadonly<T>` payloads (Task 1 design); copy
  before mutating into a host `Meeting`.
- **`@frankenstein/shared` is a devDep on purpose** ‚Äî see
  `federation.config.mjs` comment. Don't move it to
  `dependencies` without re-reading that note. Same rule
  applies to `whiteboard` and `mermaid` packages in M3/M4.
- **Workspace symlink.**
  `packages/shell/node_modules/@frankenstein/shared` ‚Üí
  `packages/shared/`. Confirmed working at runtime; same
  pattern will hold for the React/Svelte remotes.
- **Node pinned to major `22`** via `.node-version`. Any CI,
  Dockerfile, or contributor docs added in M2+ should
  reference the same.
- **`angular.json` keeps `esbuild` and `serve-original`
  targets** for non-federated debugging ‚Äî handy if M2's
  Schedule-X integration ever needs to be profiled in
  isolation from federation.
- **TS root pin is `~5.9.0`**, Angular 21's pin is `~5.9.2`.
  Don't drift below.
- **Federation init error handling is a deferred design
  decision.** `main.ts` currently logs and continues on
  init failure (schematic default). M2/M3 should split
  fatal host/manifest init from per-remote degradation,
  surface explicit error UI, and replace the inline
  comment in `main.ts` with the real implementation.
